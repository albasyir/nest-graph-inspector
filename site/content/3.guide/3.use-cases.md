## Impact Analysis

When planning a change, the dependency graph helps you understand the **blast radius** before touching any code.

- **Narrow regression test scope** to the most relevant modules and providers
- **Reduce unnecessary testing** for unrelated areas
- **Understand which downstream services** will be affected by a change

::callout{icon="i-lucide-target" color="primary"}
Before modifying a provider, check the graph to see every module and controller that depends on it.
::

## Test Prioritization

Use the graph to determine which tests matter most after a change.

- **Select critical providers** for fast validation
- **Understand dependency chains** between providers and controllers
- **Prioritize which flows** should be checked first after a change

## Architecture Visibility

The graph is a living document of your application architecture.

- **Onboard new engineers faster** — they can see the full picture instantly
- **Spot highly coupled modules** — nodes with many connections indicate tight coupling
- **Make refactors safer** — visualize relationships before moving code around

## CI/CD Integration

With JSON output, you can integrate the graph into your pipeline:

- **Compare graphs** between commits to detect structural changes
- **Alert on new circular dependencies** or unexpected module additions
- **Generate architecture reports** automatically on each deploy

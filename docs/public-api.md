# Public API Reference тАФ `nest-graph-inspector`

**Package version inspected:** 0.3.0  
**Entry point:** `nest-graph-inspector` тЖТ `dist/libs/nest-graph-inspector/src/index.js`  
**Type entry point:** `dist/libs/nest-graph-inspector/src/index.d.ts`

---

## How the public surface is defined

The package's `exports` map points a single condition (`"."`) to `index.js`/`index.d.ts`.
Everything re-exported from `index.ts` is the intended public API.

`index.ts` re-exports from exactly five source files:

```ts
export * from './nest-graph-inspector.module';   // class + const
export * from './nest-graph-inspector.type';     // config types
export * from './types/graph-output.schema';     // schema constant + object
export * from './types/graph-output.type';       // graph output types
export * from './types/direct-run.type';         // trace / direct-run types
```

The package's `files` field ships the full compiled `src/` tree, so consumers can
reach internal modules via deep imports (e.g. `nest-graph-inspector/src/adapters/тАж`).
Those deep paths are **not** in the `exports` map and are **not** part of the
supported public API.

**Intended consumers:**

- **NestJS application developers** тАФ import `NestGraphInspectorModule` to add the
  inspector to their app.
- **Tooling and CI authors** тАФ import `GraphOutput` and related types to consume or
  validate the JSON graph artifact.
- **Viewer / frontend code** тАФ imports `GraphOutput` and `DirectRun*` types via the
  `@library` alias (type-only, not runtime).

## Public entry point

### `NestGraphInspectorModule`

| | |
|---|---|
| **Kind** | Class (NestJS `@Module`) |
| **Source** | `src/nest-graph-inspector.module.ts` |
| **Stability** | Stable |
| **Documented** | Yes тАФ README, installation guide, configuration docs |

The root NestJS module consumers import. Extends `ConfigurableModuleClass`, which
adds two static factory methods generated by `ConfigurableModuleBuilder`:

| Method | Signature | Purpose |
|---|---|---|
| `NestGraphInspectorModule.forRoot(options?)` | `(options: NestGraphInspectorModuleOptions) тЖТ DynamicModule` | Configure the module with explicit options |
| `NestGraphInspectorModule.forRootAsync(options)` | `(options: ConfigurableModuleAsyncOptions<тАж>) тЖТ DynamicModule` | Configure asynchronously (e.g. from `ConfigService`) |

Using `NestGraphInspectorModule` without calling `forRoot` applies `defaultOptions`
(viewer output on port 53371 with Ollama proxy and Direct Run enabled).

**Example:**
```ts
import { NestGraphInspectorModule } from 'nest-graph-inspector';

@Module({
  imports: [NestGraphInspectorModule],           // uses defaultOptions
  // or:
  imports: [NestGraphInspectorModule.forRoot({
    outputs: [{ type: 'json', path: 'graph.json' }],
  })],
})
export class AppModule {}
```

## Exports that may not belong in the public API

The following are reachable by consumers (either via `index.ts` re-exports or via
deep imports into the shipped `src/` directory) but appear to be internal
implementation details. None have JSDoc, stability notes, or user-facing
documentation.

### `defaultOptions` (via `index.ts`)

Re-exported incidentally via `export * from './nest-graph-inspector.module'`.
It is the default configuration object used when `NestGraphInspectorModule`
is imported without `forRoot`. Consumers have no documented reason to import
or depend on this value.

**Recommendation:** Either re-export it with an explicit `export { defaultOptions }`
and add JSDoc, or stop re-exporting the module file with `export *` in favour of
an explicit named list that excludes `defaultOptions`.

---

### `NestGraphInspectorSetup` (deep import, not in `index.ts`)

The `@Injectable()` service that drives all graph extraction. It is compiled into
the dist tree and reachable at:

```
nest-graph-inspector/src/nest-graph-inspector.setup
```

It has two public methods тАФ `buildModuleMap()` and `buildModuleMapFromAutoDetect()`
тАФ that the test suite accesses, and a third (`enrichModuleMap`) that the spec
accesses only via a private-cast workaround. None of these methods carry JSDoc or
any indication that they are intended for consumers. The class is not in `index.ts`.

**Assessment:** Internal. The public methods appear to exist to satisfy testing
needs without full private casting, not to serve external consumers. If
`buildModuleMap` is intentionally public, it needs documentation, a stable type
signature, and export from `index.ts`.

---

### `RuntimeTraceRecorder` (deep import, not in `index.ts`)

The `@Injectable()` concrete implementation of `DirectRunTraceRecorder`. Compiled
into the dist and reachable at:

```
nest-graph-inspector/src/runtime-trace.recorder
```

It has two public methods beyond the interface тАФ `recordSpan()` and
`summarizeValue()`/`previewValue()` тАФ that are called internally by
`NestGraphInspectorSetup` after it instruments provider instances at runtime.
`recordSpan` takes a `RuntimeTraceSpanInput` type that is *not* exported from
`index.ts` (it is a private `type` alias inside `runtime-trace.recorder.ts`).

**Assessment:** Internal. If `RuntimeTraceRecorder` is meant to be injectable
by consumers who want to record their own spans, it needs to be added to `index.ts`
and `RuntimeTraceSpanInput` needs to be exported. As of 0.3.0 there is no
documented usage pattern for external injection.

---

### Adapter classes (deep imports, not in `index.ts`)

All six adapter classes and `HttpServeAdapter` are compiled into the dist and
reachable via deep imports:

```
nest-graph-inspector/src/adapters/http-output.adapter
nest-graph-inspector/src/adapters/http-serve.adapter
nest-graph-inspector/src/adapters/file-output.adapter

nest-graph-inspector/src/adapters/proxy.adapter

```

`HttpOutputAdapter` has a public `normalizePath()` method and a public static
`defaultConfig` property. `FileOutputAdapter` has a public `buildMarkdownText()`
method. Neither is documented for external use.

**Assessment:** Internal. These are NestJS `@Injectable()` services managed by
the module's DI container. Consumers should not inject or instantiate them
directly. If the `exports` map is ever tightened (e.g. using Node.js `exports`
subpath patterns), these should be excluded.

## Open questions

1. **`defaultOptions` intentionality** тАФ Was re-exporting `defaultOptions` a
   deliberate choice to let consumers check or extend defaults, or is it an
   accidental side-effect of `export *`?

2. **`buildModuleMap` and `buildModuleMapFromAutoDetect` on `NestGraphInspectorSetup`** тАФ
   These are public TypeScript methods with no access modifier, no JSDoc, and no
   entry in `index.ts`. Their publicity appears to be a testing convenience. Are
   they intended as a programmatic API for advanced consumers?

3. **`RuntimeTraceRecorder.recordSpan`** тАФ This method is called internally by
   `NestGraphInspectorSetup` after instrumenting provider instances. Is it intended
   to be callable by consumers who want to manually record spans inside their own
   providers?

4. **`RuntimeTraceSpanType` export** тАФ The type is used only in the internal
   `RuntimeTraceSpanInput` private type. Exporting it without also exporting
   `RuntimeTraceSpanInput` makes it hard to use. Is this export intentional?

5. **`GRAPH_OUTPUT_JSON_SCHEMA` consumer guidance** тАФ The HTTP adapter serves this
   at `/output.schema.json`, but there is no documentation explaining why a consumer
   would import it directly from the npm package vs. fetching the served URL.
